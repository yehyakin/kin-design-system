# KIN RFCs

KIN RFCs record proposals that materially change a delivery surface, generated artifact, interoperability boundary, or repository architecture.

An RFC is not a normative KIN contract. Until an RFC is accepted and its outcomes are incorporated into the governing documents, `DESIGN.md`, `DELIVERY.md`, the component and page contracts, and the release rules remain authoritative.

## Status values

| Status | Meaning |
|---|---|
| `proposed` | Ready for review; no implementation approval is implied |
| `accepted` | The direction and boundaries are approved for phased implementation |
| `implemented` | Approved work is complete and the normative sources and verification agree |
| `rejected` | The proposal will not proceed in its current form |
| `superseded` | A later RFC replaces the decision |

RFC implementation MUST remain separately reviewable. A proposal MAY include implementation details and acceptance criteria, but it MUST NOT describe unperformed work as shipped, verified, or released.

`accepted` authorizes implementation; it does not make the RFC itself normative. `implemented` requires the accepted decision to be incorporated into the governing contracts and verified.
